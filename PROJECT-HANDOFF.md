### Critical: pages can be truncated mid-footer — ROOT CAUSE FIXED 2026-06-14
**This is NOT a Google Drive / OneDrive sync problem.** Both have been disabled on this
machine for a long time. Earlier handoffs blamed Drive sync — that was wrong; do not chase it.

The real cause was the **page-generator scripts** (`gen_product_pages.py`, `gen_ind.py`,
`gen_run.py`, `gen_cities.py`, `make_og_images.py`). They wrote files with
`open(path,"w",encoding="utf-8").write(...)` — an idiom that **never explicitly closes or
flushes the file handle**. If a run was interrupted (e.g. the Cowork/bash sandbox kills a
long run, or any abnormal exit) the buffered tail never reached disk, leaving the file
**truncated near the end** — losing the footer bottom, the closing `</body></html>`, AND the
`site.v2.js` script tag. Because the footer is about the same size on every page, the cut
always landed in roughly the same spot (the footer's social-row). A truncated `index.html`
was even deployed live before it was caught (no GA4, no menu, no footer links). The
`.rstrip("\x00")` calls sprinkled through the scripts were an earlier band-aid for the same
write-corruption, not a separate issue.

**Fix (2026-06-14):** every generator now writes through a `safe_write(path, text)` helper
that uses `with open(...)` (guaranteed flush + close) and then **re-reads the file to verify
the full content landed**, aborting the run with `SystemExit` if it doesn't — so a truncated
page can never be written or deployed again. 16 unsafe writes across the 5 scripts were
converted; `build.py` already used the safe `pathlib.write_text()`.

Standing rules that still apply:
- Run the Python generators (and `checkpoint.py`) from a **real Windows terminal**, not the
  Cowork/bash sandbox — the sandbox can serve a **stale cached copy** of these files and may
  kill long runs.
- Prefer the **Read/Write/Edit file tools** over the bash sandbox for editing site files.
- Before any deploy, confirm every page still ends in `</html>` (cheap guard, see §12).

> Note: the **Workers** code lives in `C:\worker\`. Deploy Workers from a real Windows

### 5c. Anti-spam (both forms)
- **Honeypot**: hidden `<input name="_gotcha">` — a trip = silent `200` drop.
  (The contact Worker originally never checked `_gotcha` server-side; the check was added to
  `contact-form-worker.js` on 2026-07-02 — **takes effect on the next `jja-contact` deploy**.)
- **Cloudflare Turnstile**: widget sitekey `0x4AAAAAADScpMxfNdQDkoBm`.
  - Verified for **full** submissions only. **Partial leads are exempt from Turnstile**
    (honeypot still applies) so a missing/blocked token can never silently kill a lead.
  - `verifyTurnstile()` **fails OPEN** if `TURNSTILE_SECRET` is unset or Cloudflare is
    unreachable — deploying without the secret means no protection, not an outage.
    Make sure `TURNSTILE_SECRET` is set in the Worker's Cloudflare dashboard.

### 5d. Partial-lead capture (quote form)
`assets/js/site.v2.js` background-POSTs a partial lead to the Worker via
`form.dataset.worker` (JSON `{ formData: {...} }` with a `partial_lead` flag), using
`fetch(..., { keepalive: true })` so the POST survives page unload.

Triggers: **`pagehide` and `beforeunload` only** — i.e. when the visitor closes the tab
or navigates away. At most one partial per visitor. (Earlier 15-second-idle and
`visibilitychange` triggers were removed because they fired mid-typing and sent
duplicate/premature leads before the user clicked Submit.)

Validation gate: fires only when `first_name`, `last_name`, valid `email`,
`phone` (10+ digits), and `zip` (exactly 5 digits) are all filled.
Debug helper: on `/quotes/`, open DevTools console and run `window.testPartialLead()`.
